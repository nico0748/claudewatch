# Claude 複数アカウント使用量モニター — 要件定義書

**版数:** 0.2（ドラフト）
**作成日:** 2026-07-03
**更新:** 2026-07-04 データ取得方式を方式A（ブラウザセッション/内部エンドポイント参照）に確定／取得サブ方式=クッキー直接利用、ポーリング間隔=5分に確定
**対象:** 複数の Claude アカウントの使用量とリセット時刻を一元監視・通知するデスクトップアプリ

---

## 1. 目的・背景

複数の Claude アカウント（Pro / Max 等）を使い分けていると、各アカウントがそれぞれ独立した利用制限とリセットタイミングを持つため、「どのアカウントが今使えるのか」「いつリセットされるのか」を把握しづらい。本アプリは、全アカウントの使用状況とリセット時刻を常駐アプリで一元表示し、リセット予告・空きアカウント通知を行うことで、待ち時間ゼロで作業を継続できる状態をつくることを目的とする。

### 前提：Claude の利用制限の仕組み

要件の背景として、Claude Pro / Max の制限は二重構造になっている点を押さえる。

- **5時間ローリング枠**：最初のプロンプト送信時点から5時間でリセットされる。9:00に送信すれば14:00にリセット。固定時刻ではなく、使い始めを起点とする。
- **週次制限**：アカウントごとに割り当てられた固定曜日・時刻でリセット。Pro は全モデル共通で1本、Max は「全モデル共通」＋「Sonnet 系専用」の2本を持つ。
- 二つの時計が別々に進むため、片方が空いていても他方でブロックされることがある。**「複数の時計を同時に可視化する」ことが本アプリの核心的価値。**

## 2. スコープ

### 2.1 対象プラットフォーム

- **macOS**（メニューバー常駐アプリ）
- **Linux**（システムトレイ常駐アプリ / デスクトップ通知）

Windows は当面スコープ外（将来的にクロスプラットフォーム基盤で対応可能な設計とする）。

### 2.2 対象アカウント

- 複数の Claude 消費者向けアカウント（Pro / Max）を **最大4アカウント**まで監視する。
- 将来的に Anthropic API（従量課金）アカウントの併記にも拡張可能とする。

### 2.2.1 対応ブラウザ（セッション取得元）

各アカウントのセッションは、ユーザーが普段ログインしているブラウザのクッキーストアから取得する。対応ブラウザは以下4種。

- **Google Chrome**（macOS / Linux）
- **Brave**（macOS / Linux）
- **Mozilla Firefox**（macOS / Linux）
- **Safari**（macOS のみ ※Linux 版は存在しない）

各アカウントごとに「どのブラウザのセッションを使うか」を指定でき、4アカウントが別々のブラウザに紐づく構成も許容する。

### 2.3 スコープ外（初期リリース）

- 利用制限そのものの回避・自動切り替えによるプロンプト送信代行
- チーム/Enterprise 管理者向けの組織横断ダッシュボード
- Windows 対応

## 3. データ取得方式（重要論点・要検討）

Claude の**消費者向けアカウントには、使用量・リセット時刻を返す公式 API が存在しない**。このため取得方式が本プロジェクト最大の技術的リスクである。検討の結果、**方式A（ブラウザセッション/内部エンドポイント参照）を採用**する。

### ★採用：方式A — ブラウザセッション / 内部エンドポイント参照
ログイン済みセッションの資格情報（クッキー/セッショントークン）を用いて claude.ai の内部エンドポイントから、各アカウントの**使用量・枠の状態・リセット時刻を自動取得**する。
- 長所：実際の残量に近い正確な値を自動取得でき、使用量パーセンテージやリセット時刻を手入力なしで表示できる。ユーザー体験が最も良い。
- 短所（＝要対策リスク、§9で詳述）：
  - 非公開仕様のため**予告なくレスポンス構造が変わり壊れる**可能性がある。
  - **利用規約・自動化ポリシーへの抵触可能性**があり、事前確認が必要。
  - 多アカウントのセッション管理・再認証（トークン失効時）の実装が複雑。
  - CAPTCHA / bot 対策・レート制限に引っかかる可能性。

#### 方式Aの取得サブ方式 → ★確定：ブラウザのクッキーストアから直接取得
各アカウントでログイン済みのブラウザ（Chrome / Safari / Brave / Firefox）のローカルクッキーストアから claude.ai のセッションクッキーを**アプリが自動で読み出し**、そのクッキーを用いて内部エンドポイントへ直接リクエストし使用量・リセット時刻を取得する。ユーザーは手動でトークンを貼り付ける必要がない。

##### クッキー登録UX → ★決定（実装者裁量で以下を採用）
1. アプリ設定画面で「アカウントを追加」→ **使用ブラウザを選択**（インストール済みブラウザを自動検出して一覧提示）。
2. アプリが選択ブラウザのクッキーストアから claude.ai のセッションクッキーを検出。複数プロファイル/複数アカウントがある場合は候補を一覧提示し、ユーザーが対象を選択。
3. 選択したクッキーの参照情報（どのブラウザ・どのプロファイル）を保存。実クッキー値はキャッシュせず、5分ごとのポーリング時にストアから都度読み出す方式を基本とする（値の陳腐化・漏洩リスクを低減）。
4. 失効を検出した場合は該当アカウントを「要再ログイン」表示にし、ユーザーにブラウザでの再ログインを促す（再ログイン後は自動復帰）。

##### ブラウザ別クッキーストアの取得実装（技術メモ）
| ブラウザ | 保存場所/形式 | 暗号化 | 復号に必要なもの |
|----------|--------------|--------|------------------|
| Chrome / Brave | `Cookies` SQLite DB（プロファイル配下） | 値をAES暗号化 | macOS: Keychain の "Chrome/Brave Safe Storage" キー、Linux: Secret Service/kwallet または既定鍵 |
| Firefox | `cookies.sqlite`（プロファイル配下） | 値は平文 | プロファイル特定のみ |
| Safari | `Cookies.binarycookies`（macOSのみ、コンテナ配下） | 独自バイナリ形式 | パース処理（TCC のフルディスクアクセス許可が必要な場合あり） |

- クロスプラットフォーム対応：Safari は macOS 限定。Linux では Chrome/Brave/Firefox の3種を対象とする。
- 参考として検討した「手動クッキー貼り付け」「埋め込みブラウザ/拡張連携」は初期リリースでは不採用（前者は自動読み出しで不要、後者は実装コスト増のため将来オプション）。

#### ポーリング間隔 → ★確定：5分ごと
バックグラウンドで **5分間隔**で各アカウントの使用量・リセット時刻を取得・更新する。
- 規約リスク・レート制限を抑えつつ、リセット予告（既定15分前）に十分間に合う粒度。
- リセット直前など重要局面では間隔を短縮する動的ポーリングを将来オプションとして検討。
- 取得失敗時はバックオフ（間隔を段階的に延長）し、連続失効は再認証を促す。

> **設計方針：** データ取得層は抽象インターフェース化し、内部エンドポイントの仕様変更に追従しやすくする。取得失敗時は§9のフォールバック（前回値の表示＋計算ベースの推定）で表示を継続する。

### 参考：不採用とした方式
- **方式B（公式 API・従量課金）**：Pro/Max のサブスク枠とは別物のため要件を直接満たさず不採用。ただし将来 API アカウント併記（F-11）の実装基盤としては活用余地あり。
- **方式C（手動起点＋自動計算）**：規約リスクなく確実だが手入力負担があり、方式Aの取得失敗時**フォールバック手段**として位置づける。

## 4. 機能要件

### 4.1 必須機能（MVP）

| ID | 機能 | 概要 |
|----|------|------|
| F-01 | アカウント登録・管理 | 複数アカウントの追加/編集/削除。ラベル、プラン種別（Pro/Max）、週次リセット曜日・時刻を保持 |
| F-02 | 全アカウント一覧表示 | 常駐ウィンドウ/ポップオーバーで、各アカウントの状態（利用可/枠消化中/上限到達）とリセットまでの残り時間を一覧表示 |
| F-03 | リセット時刻の予告通知 | リセットの N 分前（既定15分前、ユーザー設定可）にデスクトップ通知。5時間枠・週次枠それぞれに対応 |
| F-04 | 空きアカウント通知 | いずれかのアカウントが「今すぐ使える」状態に復帰したらデスクトップ通知 |
| F-05 | 使用量の自動取得・表示 | 方式Aで各アカウントの残量%・枠状態・リセット時刻を自動取得し、バー/数値で表示 |
| F-06 | リセット時刻の取得・計算 | 内部エンドポイントから取得したリセット時刻を表示。取得不可時は5時間枠（起点+5h）/週次枠を計算しカウントダウン |
| F-07 | 上限接近アラート | 取得した使用量が閾値（既定80%、ユーザー設定可）に達したら警告通知 |
| F-08 | 常駐 UI | macOS メニューバー / Linux システムトレイに常駐。アイコンに「次のリセットまでの残り時間」または「利用可アカウント数」をバッジ表示 |
| F-09 | 認証・セッション管理 | 各アカウントのセッション資格情報を登録・保持し、失効時は再認証を促す |

### 4.2 推奨機能（Post-MVP）

| ID | 機能 | 概要 |
|----|------|------|
| F-10 | 履歴・傾向 | 各アカウントの使用量推移・リセット履歴・消化ペースの記録とグラフ |
| F-11 | クイックアクション | 一覧から該当アカウントの claude.ai を既定ブラウザで開く |
| F-12 | 手動フォールバック | 方式Aの取得失敗時、方式C（手動起点＋計算）で表示を継続 |
| F-13 | API アカウント併記 | 方式B：従量課金の当月コスト/使用量を併記 |

### 4.3 通知要件の詳細

- 通知チャネル：OS ネイティブ通知（macOS: 通知センター、Linux: `libnotify`/`notify-send` 相当）。
- 通知種別ごとに **ON/OFF とタイミングを個別設定可能**。
- 「予告通知」は枠ごと（5時間/週次）に別々の閾値を設定可能。
- 「空きアカウント通知」は連続通知の抑制（クールダウン）を行い、通知過多を防ぐ。
- 静音時間帯（Do Not Disturb 相当）の設定。

## 5. 非機能要件

| 区分 | 要件 |
|------|------|
| 常駐性 | OS 起動時に自動起動（設定でON/OFF）。バックグラウンドで低リソース動作 |
| パフォーマンス | 待機時 CPU ほぼ0%、メモリ数十MB程度を目標。ポーリング/計算は分単位で十分 |
| セキュリティ | アカウント資格情報・セッション情報は OS のセキュアストレージ（macOS Keychain / Linux Secret Service）に保存。平文ファイル禁止 |
| プライバシー | 取得データはローカル完結。外部サーバーへ送信しない |
| 可用性 | ネットワーク不通時も、計算ベースのリセット時刻表示は継続動作 |
| 保守性 | データ取得層を差し替え可能な抽象インターフェースとし、方式A/B/Cを切替・追加できる構成 |
| 国際化 | 初期は日本語UI。タイムゾーン対応必須（リセット時刻計算の正確性のため） |
| ログ | 障害調査用のローカルログ。機微情報はマスキング |

## 6. 技術構成（案）

macOS / Linux 両対応かつ常駐アプリという要件から、以下を候補とする。

- **Tauri（Rust + Web フロント）** ★確定：軽量・省メモリ、システムトレイ/メニューバー対応、セキュアストレージ連携が容易。ネイティブ通知プラグインあり。
- **Electron**：実装容易だがメモリ消費が大きく、常駐アプリには不利（不採用）。
- **ネイティブ個別実装**：最軽量だが2OS分の実装コストが高い（不採用）。

> **決定：Tauri。** バックグラウンドのデータ取得・計算層を Rust、UI をWeb技術で実装し、通知は各OSネイティブへ委譲する。
>
> ブラウザクッキーの読み出しは Rust 側で実装する。Chrome/Brave の暗号化クッキー復号（OSキーチェーン連携）、Firefox の `cookies.sqlite` 読み取り、Safari の `binarycookies` パースを担うクレート/自作モジュールを用意し、ブラウザ抽象インターフェースの背後に実装する。既存の rust-cookie 抽出系クレートの活用も検討。

### アーキテクチャ概念

```
┌─────────────────────────────────────────┐
│  常駐プロセス（トレイ/メニューバー）      │
│  ┌───────────────┐  ┌──────────────────┐ │
│  │ データ取得層   │  │ リセット計算      │ │
│  │ 方式A(内部EP) │→ │ /フォールバック   │ │
│  │ +認証/失効管理 │  │ エンジン          │ │
│  └───────────────┘  └────────┬─────────┘ │
│                              ↓            │
│  ┌───────────────┐  ┌──────────────────┐ │
│  │ 通知マネージャ │←─│ 状態ストア        │ │
│  │ (OSネイティブ) │  │ (セキュア保存)     │ │
│  └───────────────┘  └──────────────────┘ │
│              ↓ UI                         │
│      一覧ポップオーバー / 設定画面         │
└─────────────────────────────────────────┘
```

## 7. データモデル（概略）

```
Account {
  id: string
  label: string              // 表示名（例: 個人Max, 仕事Pro）
  plan: "pro" | "max"
  weekly_reset_weekday: int  // 0-6
  weekly_reset_time: time    // 固定時刻
  timezone: string
  // 5時間枠（ローリング）
  window_started_at: datetime | null   // 最初のプロンプト時刻
  window_reset_at:   datetime | null   // = started_at + 5h（計算）
  // 週次枠
  weekly_reset_at:   datetime          // 次回の週次リセット（計算）
  status: "available" | "in_window" | "limited"
  // 方式A：内部エンドポイントから取得
  usage_percent: float          // 残量/消化率
  weekly_usage_percent: float   // 週次枠の消化率（Max は複数枠）
  last_fetched_at: datetime     // 最終取得時刻
  fetch_status: "ok" | "stale" | "auth_required" | "error"
  // 認証：クッキーはブラウザストアから都度読み出すため参照情報のみ保持
  browser: "chrome" | "brave" | "firefox" | "safari"
  browser_profile: string     // 対象プロファイルの識別子
}

NotificationSetting {
  reset_advance_minutes: int  // 予告 N 分前
  notify_on_free: bool
  cooldown_minutes: int
  quiet_hours: [start, end]
  per_window_enabled: { five_hour: bool, weekly: bool }
}
```

> 最大アカウント数は **4**。上記 `Account` を最大4件保持する。

## 8. 画面要件（概略）

1. **メニューバー/トレイ ポップオーバー**：全アカウントを行で一覧。各行に「ラベル / 状態バッジ / 次のリセットまでの残り時間（5時間枠・週次枠）」。最上部に「今使えるアカウント数」。
2. **アカウント設定画面**：追加・編集フォーム。プラン、週次リセット曜日/時刻、タイムゾーン、（方式C時）枠開始の記録操作。
3. **通知設定画面**：予告分数、空き通知ON/OFF、クールダウン、静音時間帯。
4. **アイコンバッジ**：次にリセットされるアカウントの残り時間、または利用可アカウント数を表示。

## 9. 想定リスクと対応

| リスク | 影響 | 対応 |
|--------|------|------|
| 内部エンドポイントの仕様変更（方式A） | 自動取得の停止 | 取得層を抽象化し追従しやすく。取得失敗時は前回値表示＋方式C計算にフォールバック（F-12）。バージョン検知とエラー通知 |
| 利用規約・自動化ポリシー抵触（方式A） | 規約上の問題・アカウント制限リスク | **開発着手前に claude.ai 利用規約・自動化ポリシーを精査必須**。取得頻度を低く抑え（分単位）人間の閲覧に近い挙動とする。抵触が判明した場合は方式Cへ切替 |
| CAPTCHA / bot 対策・レート制限 | 取得失敗・アカウント制限 | ポーリング間隔を保守的に設定。埋め込みブラウザ方式で正規セッションを維持。バックオフ制御 |
| セッショントークンの失効・多アカウント管理（最大4） | 取得停止・再ログイン負担 | 失効検知→「要再ログイン」表示（F-09）。ブラウザで再ログインすれば自動復帰 |
| ブラウザ暗号化クッキーの復号権限 | Chrome/Brave のクッキー復号にOSキーチェーンアクセスが必要。macOSでKeychain許可ダイアログが出る | 初回のみユーザーに許可を案内。Linux は Secret Service/kwallet 連携、非対応環境は既定鍵で復号 |
| Safari クッキーのアクセス権限 | macOS の TCC によりフルディスクアクセス許可が必要な場合がある | 初回セットアップで許可手順をガイド表示 |
| ブラウザのクッキー暗号化仕様変更 | クッキー読み出し失敗 | ブラウザ抽象層で吸収。失敗時は該当アカウントをエラー表示しフォールバック |
| 読み出したクッキーの漏洩 | セキュリティ事故 | クッキー値はキャッシュせず都度読み出し。メモリ上のみで扱い外部送信なし。参照情報のみOSセキュアストレージに保存 |
| タイムゾーン/夏時間 | リセット時刻の誤差 | タイムゾーン明示保持、テストケース整備 |
| 通知過多 | ユーザー体験悪化 | クールダウン・静音時間帯・枠別ON/OFF |
